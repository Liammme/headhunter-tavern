# 赏金池整体架构优化说明

## 1. 文档目的

本文档用于定义 `赏金池` 当前版本的整体架构优化方向。

目标不是推倒重做，而是把现在已经跑通的产品，从“能跑的单体闭环”优化成一套：

1. 低耦合
2. 职责清晰
3. 可解释
4. 可回放
5. 可渐进演进

的产品架构。

一句话概括：

**把当前“抓取后直接在线拼首页”的实现，逐步升级成“事实层 + 分析层 + 展示层”三段式架构。**

---

## 2. 当前系统的主要问题

当前版本已经具备完整主链路：

1. 抓取岗位
2. 入库
3. 首页展示
4. 岗位认领
5. 顶部情报占位

但从架构角度看，当前系统最大的问题不是“缺功能”，而是：

**多个层级的职责混写在同一批 service 里。**

### 2.1 当前耦合最重的模块

#### `crawl_pipeline.py`

当前这个文件同时负责：

1. 抓取调度
2. 去重
3. 标准化
4. 字段识别
5. 标签生成
6. 评分
7. 入库
8. 过期清理

它更像一个“小型总控器”，而不是清晰的流水线。

#### `home_feed.py`

当前这个文件同时负责：

1. 查库
2. 认领合并
3. 时间窗口过滤
4. 今天 / 昨天 / 更早分桶
5. 公司聚合
6. 公司等级推断
7. 岗位排序
8. 首页 JSON 组装

这意味着首页展示层正在承担业务聚合与排序判断。

#### `claims.py`

当前认领逻辑还在 API 层里直接落库，说明“认领”尚未成为独立应用能力。

#### `intelligence.py`

当前顶部 `猎场情报` 还主要是静态占位文案，没有真正建立在统一的数据分析基线上。

### 2.2 当前问题的本质

当前系统更像：

**抓取时顺手做分析，展示时再补一轮聚合。**

这会导致：

1. 改抓取逻辑时容易碰到评分和首页逻辑
2. 改评分规则时容易污染入库逻辑
3. 改首页排序时容易重新拼业务规则
4. 顶部情报和下面榜单可能来自不同判断基线

---

## 3. 整体优化目标

这轮架构优化的目标是：

1. 保持现有系统继续可用
2. 把抓取、评分、首页、认领、情报逐步拆成独立层级
3. 让首页消费稳定的读模型，而不是每次请求都临时拼复杂逻辑
4. 让评分从硬编码规则升级成可解释、可回放、可迭代的决策能力
5. 让 `猎场情报` 与岗位列表共享同一分析基线
6. 支持未来新增来源、调整评分、优化首页，而不牵动整条链

最终目标系统应具备以下特征：

1. 新接一个来源，不需要改首页
2. 调一条评分规则，不需要重写 crawler
3. 改情报策略，不会打坏岗位池
4. 首页只消费稳定 DTO，不再自己重算业务规则
5. 评分结果可解释、可版本化、可回放

---

## 4. 核心设计原则

### 4.1 契约优先

优先稳定以下对外契约：

1. `GET /api/v1/home`
2. `POST /api/v1/claims`
3. `POST /api/v1/crawl/trigger`

内部实现可以演进，但前后端契约不要频繁漂移。

### 4.2 读写分离

把系统明确分为：

1. 写路径：抓取、标准化、事实提取、评分、快照生成
2. 读路径：首页、公司卡、认领展示、情报展示

写路径负责“形成结果”，读路径负责“消费结果”。

### 4.3 单职责

每一层只回答一个问题：

1. 抓取层：抓到了什么
2. 标准化层：如何统一表示
3. 事实提取层：识别到了什么事实
4. 评分层：这些事实意味着什么
5. 排序/聚合层：首页应该怎么排
6. 情报层：最近市场发生了什么
7. 认领层：谁认领了什么
8. 首页组装层：前端拿到什么 JSON

### 4.4 兼容优先

新能力应优先采取：

1. 旁路实现
2. 双写兼容
3. 逐步切换

而不是一次性替换旧链路。

### 4.5 可回滚

每次架构优化都应保证：

1. 首页至少还能按旧逻辑跑
2. 评分至少还能回退到旧版本
3. 抓取失败时不会导致整站空白

### 4.6 可解释

以下输出必须可解释：

1. `高赏金 / 中赏金 / 低赏金`
2. `重点公司 / 关注公司 / 普通公司`
3. `猎场情报`

系统输出不能只是黑箱结论。

### 4.7 数据先行

先把：

1. 原始事实
2. 派生分析结果
3. 展示快照

三层数据边界理清，再谈更复杂的推荐与策略。

---

## 5. 推荐的整体分层架构

建议把后端明确分成 8 层。

### 5.1 抓取层

职责：

1. 调用 source adapters
2. 从站点拿到岗位

输入：

1. 源配置
2. 站点地址
3. 抓取策略

输出：

1. `NormalizedJob`

抓取层不应该依赖：

1. 评分规则
2. 首页排序
3. 认领逻辑
4. 情报逻辑

### 5.2 标准化层

职责：

1. 把不同来源统一成同一 canonical 结构
2. 统一处理 company、url、时间等基础字段

输入：

1. `NormalizedJob`

输出：

1. 标准岗位输入对象

标准化层不应该依赖：

1. 评分
2. 首页标签
3. 公司卡等级

### 5.3 事实提取层

职责：

1. 从标准岗位中提炼结构化事实

输入：

1. 标准岗位对象

输出示例：

1. `job_category`
2. `domain_tag`
3. `seniority`
4. `is_urgent`
5. `has_hiring_anomaly`
6. `is_bd_entry`
7. `fact_tags`

事实提取层不应该依赖：

1. 页面文案
2. 首页排序逻辑
3. 分数阈值

### 5.4 评分层

职责：

1. 基于事实计算岗位分数和等级
2. 输出解释原因

输入：

1. 岗位事实

输出：

1. `score`
2. `bounty_grade`
3. `rule_hits`
4. `reasons`

评分层应尽量保持纯函数特性，不依赖：

1. DB
2. API
3. crawler adapter
4. 首页结构

### 5.5 聚合 / 排序层

职责：

1. 日期分桶
2. 公司聚合
3. 公司等级生成
4. 岗位排序
5. 公司排序

输入：

1. 岗位评分结果
2. 认领事实

输出：

1. 公司卡读模型
2. 日级分组结果

### 5.6 认领层

职责：

1. 创建认领
2. 聚合认领状态
3. 输出岗位和公司层的认领视图

输入：

1. `job_id`
2. `claimer_name`

输出：

1. claim 事实
2. claim 聚合结果

认领层不应该依赖：

1. 评分
2. crawler
3. 首页内部结构

### 5.7 情报层

职责：

1. 基于近 14 天分析结果与历史轻量统计生成 `猎场情报`

输入：

1. 岗位分析结果
2. 公司聚合结果
3. 历史统计

输出：

1. `IntelligenceSnapshot`

情报层不应该直接依赖：

1. crawler adapter
2. 原始抓取结果
3. 首页 payload 结构

### 5.8 首页组装层

职责：

1. 把公司卡、认领信息、情报快照组装成前端需要的 `HomePayload`

输入：

1. 公司卡读模型
2. 情报快照
3. 认领聚合结果

输出：

1. 首页 JSON

首页组装层不应该：

1. 再次进行业务评分
2. 再次进行字段识别
3. 直接碰 crawler

---

## 6. 建议的依赖方向

推荐依赖方向如下：

1. `抓取层 -> 标准化层`
2. `标准化层 -> 事实提取层`
3. `事实提取层 -> 评分层`
4. `评分层 -> 聚合 / 排序层`
5. `聚合 / 排序层 -> 首页组装层`

认领层与情报层是两条平行支线：

1. 认领层向聚合层提供认领事实
2. 情报层读取聚合结果和历史数据生成情报快照

首页最终消费：

1. 公司聚合结果
2. 认领结果
3. 情报快照

---

## 7. 明确的依赖红线

以下边界建议作为架构红线：

1. 首页组装层不能直接依赖抓取层
2. 首页组装层不能自己重新算评分
3. 评分层不能依赖 DB、API、crawler adapter
4. 标准化层不能知道前端标签如何展示
5. 事实提取层不能混入 UI 文案
6. 认领层不能为了首页方便去重写首页聚合逻辑
7. 情报层不能直接读取 crawler 返回值
8. crawler 不能直接依赖评分规则

一句话：

**上游只提供事实，下游只消费结果，不允许跨层回头做判断。**

---

## 8. 推荐的数据分层

建议把数据明确分为 3 类。

### 8.1 原始事实数据

回答：

**抓到了什么**

示例：

1. `Job`
2. `JobClaim`
3. 抓取运行元数据

原始事实应尽量保留可追溯字段，例如：

1. `source_job_id`
2. `canonical_url`
3. `source_name`
4. `raw_payload`
5. `first_seen_at`
6. `last_seen_at`
7. `crawl_run_id`

### 8.2 派生分析数据

回答：

**这些事实在当前规则版本下意味着什么**

示例：

1. `job_category`
2. `domain_tag`
3. `signal_tags`
4. `score`
5. `bounty_grade`
6. `company_grade`
7. `rule_version`
8. `reasons`

### 8.3 展示快照 / 读模型

回答：

**首页今天应该怎么展示**

示例：

1. `CompanyDailySummary`
2. `IntelligenceSnapshot`
3. 后续可选的 `HomeFeedSnapshot`

## 9. 现有模型的职责重整建议

### 9.1 Job

`Job` 建议尽量回归成事实表。

当前问题：

1. `job_category`
2. `domain_tag`
3. `bounty_grade`
4. `signal_tags`

这些都更接近派生分析结果，而不是原始事实。

建议后续增加的事实字段：

1. `source_job_id`
2. `raw_payload`
3. `first_seen_at`
4. `last_seen_at`
5. `crawl_run_id`
6. `content_hash`

### 9.2 JobClaim

当前职责相对清楚，属于认领事实表。

后续可增强：

1. 去重策略
2. 撤销 / 转移
3. 备注 / 来源
4. 轻量状态控制

### 9.3 CompanyDailySummary

当前应该被明确成：

**公司日级读模型 / 公司聚合快照**

建议增加：

1. `analysis_version`
2. `rule_version`
3. `window_start`
4. `window_end`
5. `generated_at`

### 9.4 IntelligenceSnapshot

当前应该被明确成：

**首页顶部情报快照**

建议增加：

1. `snapshot_date`
2. `analysis_version`
3. `rule_version`
4. `window_start`
5. `window_end`
6. `generated_at`

## 10. 建议新增的中间结构

### 10.1 JobFactSnapshot

用途：

存“岗位事实快照”，回答：

**这个岗位被识别出了哪些事实**

示例字段：

1. `job_id`
2. `job_category`
3. `domain_tag`
4. `seniority`
5. `is_urgent`
6. `has_hiring_anomaly`
7. `is_bd_entry`
8. `fact_tags`
9. `fact_confidence`
10. `extracted_at`

### 10.2 JobAnalysisSnapshot

用途：

存“某一版规则下的岗位分析结果”，回答：

**在当前规则版本下，这个岗位的价值判断是什么**

示例字段：

1. `job_id`
2. `rule_version`
3. `score`
4. `bounty_grade`
5. `rule_hits`
6. `reasons`
7. `generated_at`

### 10.3 过渡策略

如果短期不想新增表，可以先：

1. 在服务层引入 DTO
2. 把 v2 元数据先放进 `Job.signal_tags`

但这只应作为过渡，不建议长期依赖一张表承载所有层次信息。

## 11. 首页和猎场情报的统一分析基线

首页与 `猎场情报` 必须基于同一分析批次或同一分析窗口。

推荐做法：

1. 每次抓取后生成一个 `analysis_run`
2. 这次 run 统一产出：
   - 岗位分析结果
   - 公司日级聚合结果
   - 顶部情报快照

这样：

1. 首页列表读同一批分析结果
2. 顶部情报读同一批趋势结果

好处是：

1. 情报和榜单不再各算各的
2. 可以追踪某一天的完整分析基线
3. 后续做回放和验证更容易

## 13. 分阶段优化路线

### 13.1 阶段 0：稳定现状

目标：

1. 冻结 API 契约
2. 补关键回归测试
3. 明确去重、唯一键、失败回退

交付：

1. 当前系统可控
2. 当前首页、认领、抓取不再脆弱漂移

### 13.2 阶段 1：拆职责

目标：

1. 从 `crawl_pipeline.py` 拆出标准化与事实提取
2. 从 `home_feed.py` 拆出排序与组装
3. 从 API 中抽出 `claim_service`

交付：

1. 抓取、评分、聚合、认领职责清楚

### 13.3 阶段 2：建立分析层

目标：

1. 引入 v2 评分并行能力
2. 入库时保留原始事实与分析结果
3. 首页仍兼容旧字段

交付：

1. 可解释评分
2. 可回放分析结果

### 13.4 阶段 3：读模型化

目标：

1. `CompanyDailySummary` 正式承接首页
2. `IntelligenceSnapshot` 正式承接顶部情报
3. 首页以快照优先，实时逻辑兜底

交付：

1. 首页与情报共享同一分析基线

### 13.5 阶段 4：回填与灰度验证

目标：

1. 对已有岗位重评分
2. 做 v1 / v2 Top 20 对比
3. 人工验证后切默认

交付：

1. backfill 能力
2. 回滚能力
3. 排名质量验证机制

---

## 14. 每阶段的主要风险

### 14.1 首页契约被破坏

风险：

前端当前直接依赖 `days / intelligence / claimed_names / jobs` 结构，内部重构时最容易误伤。

规避：

1. 先保字段
2. 再改内部实现
3. 必要时保兼容字段

### 14.2 重复抓取和重复入库

风险：

多源抓取时，canonical_url、公司标准化不一致，会导致重复岗位或脏数据。

规避：

1. 统一 canonical key
2. 统一 company normalization
3. 保持幂等 upsert
4. 补唯一约束

### 14.3 评分漂移

风险：

评分规则一旦改得太快，高赏金会失真。

规避：

1. 规则分层
2. 版本化
3. A/B 对比
4. Top 20 人工评审

### 14.4 情报空心化

风险：

顶部情报如果长期只是模板文案，很快失去价值。

规避：

1. 先做结构化快照
2. 再做生成表达
3. 与首页共享分析基线

### 14.5 一次性迁移过大

风险：

最危险的是“一步到位式重构”。

规避：

1. 每一步保旧逻辑可用
2. 新能力先双写 / 旁路
3. 读模型替换优先于全链路重写

---

## 15. 最终系统形态

最终应该得到一个四层清晰的内部猎头工作台：

1. `采集层`
   负责多源抓取、标准化、去重、入库

2. `决策层`
   负责岗位评分、公司聚合、情报生成

3. `读模型层`
   负责公司卡、日分组、认领汇总、情报快照

4. `展示层`
   负责前端渲染和轻交互，不承担复杂业务判断

最终系统应具备：

1. 新接一个来源，不用改首页
2. 调一条评分规则，不用改认领逻辑
3. 改情报策略，不会打坏岗位列表
4. 首页继续保持轻、快、可行动
5. 整个链路从“能跑”升级成“能长期维护、能持续进化、能安全重构”

---

## 16. 一句话总结

这轮优化的本质不是“把代码拆得更散”，而是：

**让抓取负责拿数据，事实层负责理解数据，分析层负责判断价值，展示层负责输出产品。**

只有这样，`赏金池` 才能从一个能跑的 MVP，演进成一个真正能长期支撑产品与业务迭代的系统。
