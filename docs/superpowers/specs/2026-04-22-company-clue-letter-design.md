# 单公司线索来信设计

日期：2026-04-22  
项目：`F:\赏金猎人`  
范围：公司卡上的单公司 LLM 线索分析能力  
不含：完整线索系统、额外联网搜索、联系人深爬、数据库迁移

## 1. 目标

本功能的目标不是建设一个完整的 BD 线索数据库，而是提供一个**按需触发的单公司线索研判能力**。

用户在浏览公司猎单池时，往往会遇到这样的问题：

- 这家公司值不值得切？
- 现有信息里，哪部分最值得先看？
- 这家公司更像补位、扩张，还是试探性招聘？
- 如果现在就开始做 BD，第一步应该从哪里下手？

当前系统已经拥有：

- 公司级聚合
- 重点岗位摘要
- 评分与标签
- 官网 / 招聘页 / 原帖 / 邮箱等低成本入口（部分来源）
- LLM 首页情报能力

但这些信息仍然是分散的，不能直接帮助猎头形成“下一步怎么动”的判断。

因此，本功能的目标是：

**点击一次按钮，生成一封由 `James 侦探` 写给你的单公司线索来信。**

## 2. 功能定位

这不是首页主模块，也不是常驻详情面板。  
它是公司卡上的一个按需触发动作。

它回答的问题不是：

- “这个岗位分数高不高”

而是：

- “针对这家公司，当前手里的线索意味着什么”
- “这家公司看起来像什么状态”
- “如果现在要动，最应该先从哪儿切”

## 3. 交互形态

### 3.1 入口

每张公司卡上提供一个按钮：

- `侦查线索`

它不属于首页主情报的一部分，而属于单公司下钻动作。

### 3.2 触发流程

用户点击 `侦查线索` 后：

1. 按钮进入 loading
2. loading 文案轮播显示
3. 生成完成后弹出一张卡片式便笺

### 3.3 Loading 文案

loading 使用三段轮播文案，按顺序切换，每段停留约 1.5 到 2 秒：

1. `James 正在翻这份档案……`
2. `别催，大侦探正在看线索。`
3. `正在整理这家公司的可用入口……`

目的：

- 保留角色感
- 同时明确告诉用户：系统正在做线索整理，而不是普通加载

## 4. 卡片形式

### 4.1 视觉定位

不是正式分析报告，不是系统弹窗，也不是详情页。

它应呈现为：

**James 临时写给你的手写便笺。**

### 4.2 视觉特征

- 比首页主情报纸页更轻、更私人
- 手写纸条感
- 轻度档案纸张材质
- 有角色语气
- 有手写签名落款

这张卡片像是从档案堆里抽出来、递到你手边的一张情报纸条。

## 5. 权限与触发规则

### 5.1 谁可以点

所有公司都可以点 `侦查线索`。

不要求先认领公司。

这样设计的原因是：

- 用户可以先侦查，再决定是否认领
- 更符合“先判断，再下手”的产品逻辑

### 5.2 缓存策略

本功能第一版不做缓存。

即：

- 每次点击都重新生成
- 不复用上一次结果

这样做的原因：

- 语义更像一次现场研判
- 不需要先设计缓存、失效和刷新逻辑
- 保持功能边界简单

## 6. 内容结构

来信长度控制在 **300-500 字**，结构固定为三块。

### 6.1 我先看到的

这部分负责把系统手头已有的线索说清楚。

内容可包括：

- 公司主页
- 招聘页
- 原帖
- 邮箱（若有）
- 重点岗位
- 评分理由
- 时间压力 / 招聘异常 / 岗位组合

它回答的是：

**“你手上现在到底有什么牌。”**

### 6.2 这说明什么

这部分体现 `James 侦探` 的推理判断。

内容应围绕：

- 这家公司更像补位、扩张，还是试探性招聘
- 哪类岗位暴露了真实需求
- 当前线索足不足以支撑立即切入

推理感要明显，但表达必须直接，不写成逻辑题或谜语。

它回答的是：

**“这些线索合起来，说明这家公司现在到底是什么状态。”**

### 6.3 你下一步怎么动

这部分给方向，不给硬命令。

内容可以包括：

- 先从官网还是招聘页切
- 哪类岗位更适合作为切口
- 当前没有直接触达线索时，先补哪一类入口

它回答的是：

**“如果你现在要动，先从哪儿下手更稳。”**

## 7. 语气设计

### 7.1 角色

固定角色为：

- `James 侦探`

### 7.2 开场

开场带角色感，但必须很短，不拖沓。

推荐风格：

- `啧，又来找大侦探我干什么？`
- `行吧，我替你翻了翻这家的线索。`

### 7.3 整体语气

要求：

- 有一点人物感
- 有一点推理感
- 信息表达直接
- 不写成谜语
- 不写成报告
- 不写成任务清单

角色感和信息密度比例建议为：

- 角色感 30%
- 信息与判断 70%

## 8. 输入边界

第一版只允许使用系统已有内容，不额外联网搜索，不深爬。

可输入内容包括：

- 公司名
- 公司链接
- 招聘页
- 原帖
- 邮箱（若有）
- 该公司重点岗位
- 标签
- v2 评分理由与命中项
- 时间压力 / 异常 / 关键性信号
- 预计赏金（若有）

### 8.1 不做项

明确不做：

- 额外全网搜索
- 猜测官网 / 猜测社媒
- 深爬联系人
- 自动补 LinkedIn / X / Telegram / Discord
- 完整联系人库建设

本功能只负责：

**把当前已有信息整理成一封对 BD 有帮助的来信。**

## 9. 失败策略

生成失败时：

- 直接提示失败
- 不生成规则版伪装来信
- 不混入 fallback 内容

原因：

- 首页 `猎场情报` 可以接受 fallback
- 但单公司 `侦查线索` 是一次高预期动作
- 如果返回一封假的“规则版来信”，会破坏功能语义

## 10. 与现有系统的关系

### 10.1 与首页主情报的关系

- 首页主情报负责宏观判断
- 单公司线索来信负责单点下钻

两者语气统一，但层级不同：

- 首页主情报：今日猎场总判断
- 单公司来信：这家公司现在怎么切

### 10.2 与公司级认领的关系

`侦查线索` 不依赖认领状态。

用户可以：

1. 先点 `侦查线索`
2. 看完来信
3. 再决定是否认领公司

### 10.3 与未来线索系统的关系

本功能不等于完整线索系统，但可以作为未来线索系统的前置验证器。

它的价值在于先回答：

**“系统已有信息，经 LLM 整理后，是否已经对 BD 真有帮助。”**

如果答案是肯定的，再考虑把其中稳定的部分结构化沉淀成正式线索字段。

## 11. 本轮不做项

本轮明确不包含：

- 结构化 BD 线索数据库
- 联系人深爬
- 公司级线索历史缓存
- 多轮对话式侦查
- 批量公司线索分析
- 将来信结果写回首页
- 前端复杂详情页

## 12. 成功标准

本功能上线后，应达到：

1. 用户点击 `侦查线索` 后，能明显感受到这是一次“侦探来信”
2. 来信能清楚说出当前已有线索
3. 来信能给出可理解的推理判断
4. 来信最后能给一个现实、保守但有用的下一步方向
5. 用户读完后，能更快决定这家公司是否值得认领和从哪里下手
