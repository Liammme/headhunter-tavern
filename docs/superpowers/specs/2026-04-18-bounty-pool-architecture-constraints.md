# 赏金池架构约束文档

## 1. 文档目的

本文档定义 `赏金池` 后续演进必须遵守的架构约束。

它不是一份实现方案，也不是一份功能规划，而是一份：

1. 分层边界约束
2. 依赖方向约束
3. 数据职责约束
4. API 与读模型约束
5. 测试与迁移约束

的正式工程规范。

一句话概括：

**用约束保证架构清晰和低耦合，防止系统在后续开发中重新缠回去。**

---

## 2. 适用范围

本约束文档适用于：

1. 抓取相关改动
2. 评分相关改动
3. 首页与公司卡改动
4. 认领机制改动
5. `猎场情报` 改动
6. 数据模型与读模型改动
7. 所有新的 service、DTO、snapshot、pipeline 设计

所有新代码和重构都应优先服从本文档，而不是局部方便。

---

## 3. 核心目标

本系统的目标架构必须满足：

1. `低耦合`
2. `职责单一`
3. `读写分离`
4. `规则可解释`
5. `结果可回放`
6. `迁移可回滚`

不满足这些目标的设计，即使短期实现方便，也不应成为正式实现方向。

---

## 4. 分层定义

系统后端被定义为以下 8 层：

1. `抓取层`
2. `标准化层`
3. `事实提取层`
4. `评分层`
5. `聚合 / 排序层`
6. `认领层`
7. `情报层`
8. `首页组装层`

所有新增逻辑必须明确属于其中一层，不允许“模糊归属”。

---

## 5. 当前仓库的高风险耦合热点

下列文件是当前仓库里最容易破坏低耦合约束的位置，后续改造必须优先受本约束文档约束：

1. `backend/app/services/crawl_pipeline.py`
2. `backend/app/services/home_feed.py`
3. `backend/app/services/scoring.py`
4. `backend/app/api/claims.py`
5. `backend/app/services/intelligence.py`
6. `frontend/components/CompanyCard.tsx`
7. `frontend/components/IntelligencePanel.tsx`
8. `frontend/lib/api.ts`

这些文件的风险不是“不能改”，而是：

1. 容易继续塞入跨层逻辑
2. 容易把临时兼容代码固化成正式结构
3. 容易让前端展示层和后端分析层互相污染

因此，凡是改到这些文件时，必须额外回答：

1. 这次改动是在拆耦合，还是在继续叠耦合？
2. 这段逻辑是否属于该文件所在层？
3. 是否存在更合适的中间层或 service 承接？

---

## 6. 当前模块归属建议

为避免后续“看哪里方便就往哪里塞逻辑”，当前仓库的主要模块应按下列归属理解：

1. `backend/app/crawlers/**` 属于 `抓取层`
2. `backend/app/services/crawl_pipeline.py` 当前是 `抓取编排层` 的过渡实现，不得继续承载评分、首页排序、情报语义
3. `backend/app/services/scoring.py` 属于 `评分层`
4. `backend/app/services/home_feed.py` 当前是 `聚合 / 排序层 + 首页组装层` 的过渡实现，后续必须拆开
5. `backend/app/services/intelligence.py` 属于 `情报层`
6. `backend/app/api/home.py`、`backend/app/api/crawl.py`、`backend/app/api/claims.py` 属于 `API 层`
7. `frontend/components/**` 属于 `展示层`，不得反向承接业务规则

如果新增模块，必须在设计或评审时明确写出其层级归属。

---

## 7. 分层职责约束

### 7.1 抓取层

抓取层职责仅限于：

1. 调用 adapters
2. 访问外部站点
3. 解析站点数据
4. 产出 `NormalizedJob`

抓取层必须：

1. 只关注如何拿到岗位
2. 保持对业务规则无感知

抓取层禁止：

1. 直接计算赏金等级
2. 直接计算公司等级
3. 直接决定首页排序
4. 依赖认领逻辑
5. 依赖情报逻辑

### 7.2 标准化层

标准化层职责仅限于：

1. 统一字段结构
2. 清洗基础数据
3. 规范 canonical url
4. 规范公司名与基础时间字段

标准化层必须：

1. 输出统一的标准岗位输入对象
2. 与具体来源站点细节解耦

标准化层禁止：

1. 生成 UI 标签
2. 直接打分
3. 生成首页展示字段
4. 依赖首页与情报需求

### 7.3 事实提取层

事实提取层职责仅限于：

1. 从标准岗位中提取结构化事实

可提取内容包括：

1. `job_category`
2. `domain_tag`
3. `seniority`
4. `is_urgent`
5. `has_hiring_anomaly`
6. `is_bd_entry`
7. `fact_tags`

事实提取层必须：

1. 只输出“事实信号”
2. 不输出最终业务等级

事实提取层禁止：

1. 直接写 `high / medium / low`
2. 混入前端文案
3. 混入首页排序逻辑

### 7.4 评分层

评分层职责仅限于：

1. 根据事实计算岗位分数与等级
2. 输出解释原因与规则命中结果

评分层必须：

1. 基于事实输入工作
2. 尽量保持纯函数
3. 支持版本化
4. 可解释

评分层禁止：

1. 直接操作数据库
2. 直接访问 API
3. 直接读取 crawler adapter
4. 直接拼首页 payload

### 7.5 聚合 / 排序层

聚合 / 排序层职责仅限于：

1. 日期分桶
2. 公司聚合
3. 公司卡等级生成
4. 岗位排序
5. 公司排序
6. 读模型生成

聚合 / 排序层必须：

1. 基于评分结果与认领事实工作
2. 只输出读模型，不直接输出前端结构细节

聚合 / 排序层禁止：

1. 重新做字段识别
2. 重新做岗位评分
3. 直接调用 crawler

### 7.6 认领层

认领层职责仅限于：

1. 创建认领
2. 聚合认领状态
3. 提供岗位 / 公司维度认领结果

认领层必须：

1. 保持独立
2. 只输出认领事实与认领视图

认领层禁止：

1. 直接计算首页排序
2. 修改评分逻辑
3. 为了页面方便而侵入聚合逻辑

### 7.7 情报层

情报层职责仅限于：

1. 基于近 14 天分析结果与历史统计生成 `猎场情报`

情报层必须：

1. 基于统一分析基线工作
2. 可降级
3. 可版本化

情报层禁止：

1. 直接依赖 crawler adapter
2. 直接读取原始抓取结果后自行做一套独立判断
3. 反向定义首页排序

### 7.8 首页组装层

首页组装层职责仅限于：

1. 组装 `HomePayload`

首页组装层必须：

1. 只消费聚合结果、认领结果、情报快照
2. 保持对前端契约稳定

首页组装层禁止：

1. 重新做评分
2. 重新做事实提取
3. 直接操作数据库进行复杂业务计算

---

## 8. 依赖方向约束

### 8.1 唯一允许的主方向

允许的主依赖方向只有：

1. `抓取层 -> 标准化层`
2. `标准化层 -> 事实提取层`
3. `事实提取层 -> 评分层`
4. `评分层 -> 聚合 / 排序层`
5. `聚合 / 排序层 -> 首页组装层`

辅助依赖方向：

1. `认领层 -> 聚合 / 排序层`
2. `聚合 / 排序层 -> 情报层`
3. `情报层 -> 首页组装层`

### 8.2 明确禁止的跨层依赖

以下依赖必须禁止：

1. 首页组装层直接依赖抓取层
2. 首页组装层直接依赖评分层内部实现
3. 评分层直接依赖 crawler adapter
4. 评分层直接依赖 API 请求结构
5. 情报层直接依赖抓取层原始输出
6. 认领层直接改首页聚合逻辑
7. 抓取层直接依赖评分规则
8. 标准化层直接依赖前端标签定义

### 8.3 例外规则

只有以下情况允许临时跨层：

1. 为兼容旧链路而做的短期过渡逻辑
2. 必须保证现有系统不宕机的应急兼容代码

即使存在例外，也必须满足：

1. 有清晰注释
2. 有明确移除计划
3. 不得成为长期默认结构

---

## 9. 数据与模型约束

### 9.1 数据三分原则

系统数据必须明确分成三类：

1. `原始事实数据`
2. `派生分析数据`
3. `展示快照 / 读模型`

任何模型都必须明确属于其中之一。

### 9.2 原始事实模型约束

原始事实模型回答：

**抓到了什么**

例如：

1. `Job`
2. `JobClaim`
3. 抓取运行元数据

原始事实模型必须：

1. 尽量保持可追溯
2. 保留来源信息
3. 保留事实本身，不混入展示快照字段

原始事实模型禁止：

1. 长期混入页面专用字段
2. 作为唯一评分存储载体
3. 在清理策略中直接破坏回放能力

### 9.3 派生分析模型约束

派生分析模型回答：

**这些事实在当前规则版本下意味着什么**

例如：

1. `job_category`
2. `domain_tag`
3. `score`
4. `bounty_grade`
5. `company_grade`
6. `rule_version`
7. `reasons`

派生分析模型必须：

1. 支持规则版本化
2. 支持重算
3. 与原始事实解耦

### 9.4 展示快照模型约束

展示快照回答：

**首页今天应该怎么展示**

例如：

1. `CompanyDailySummary`
2. `IntelligenceSnapshot`
3. 后续可扩展的 feed snapshot

展示快照必须：

1. 带生成时间
2. 带窗口信息
3. 带分析版本

展示快照禁止：

1. 反向覆盖原始事实
2. 被当成唯一事实来源

### 9.5 Job 模型约束

`Job` 应尽量回归为事实表。

以下内容长期不应只存在于 `Job` 中作为唯一真相：

1. `bounty_grade`
2. `company_grade`
3. `首页排序键`
4. `解释标签`

如果短期需要兼容，可暂放在 `signal_tags`，但必须视为过渡。

### 9.6 事实与分析快照建议

建议新增以下中间层结构：

1. `JobFactSnapshot`
2. `JobAnalysisSnapshot`

如果短期不新增表，则至少必须：

1. 引入服务层 DTO
2. 保留 `rule_version`
3. 保留 `analysis_version`

---

## 10. API 约束

### 10.1 API 层职责

API 层只负责：

1. 请求校验
2. 调用应用服务
3. 返回稳定 DTO

API 层禁止：

1. 写复杂业务规则
2. 直接做多层聚合
3. 直接内联评分逻辑
4. 直接承担认领业务规则

### 10.2 首页 API 约束

`GET /api/v1/home` 必须：

1. 依赖稳定的聚合结果和情报快照
2. 尽量不直接从原始 `jobs` 表即时拼复杂逻辑

除非作为兼容兜底，否则不允许把首页长期建立在“即时现算”上。

### 10.3 认领 API 约束

`POST /api/v1/claims` 必须通过独立认领服务执行。

禁止：

1. 在 API handler 中直接堆积业务判断
2. 因前端页面需要而把认领逻辑写死在 handler 中

### 10.4 抓取 API 约束

`POST /api/v1/crawl/trigger` 只允许触发抓取与后续分析编排，不允许在 handler 中直接塞入临时业务规则。

---

## 11. 首页与情报统一基线约束

首页列表与 `猎场情报` 必须共享同一分析基线。

这意味着：

1. 同一批抓取 / 分析结果应对应同一个 `analysis_run`
2. 首页读模型与情报快照应来自同一窗口、同一规则版本

禁止：

1. 首页用一套规则
2. 情报再临时读原始 jobs 自己算另一套结论

如果首页和情报结论来源不一致，则视为违反架构约束。

---

## 12. 测试分层约束

### 12.1 测试必须按层设计

建议测试至少分为四层：

1. `事实提取测试`
2. `评分规则测试`
3. `聚合 / 排序测试`
4. `编排 / 集成测试`

### 12.2 事实提取测试约束

事实提取测试只验证：

1. 文本、URL、时间如何被识别成结构化事实

不允许在这个层级测试里直接断言：

1. `high / medium / low`

### 12.3 评分测试约束

评分测试只验证：

1. 已知事实输入下的分数、等级、原因

评分测试不应依赖：

1. crawler adapter
2. API 层
3. 首页 payload

### 12.4 聚合 / 排序测试约束

聚合测试只验证：

1. 岗位排序
2. 公司卡等级
3. 公司卡排序
4. 分桶逻辑

### 12.5 集成测试约束

集成测试只验证：

1. 整条链是否打通
2. 数据是否落到正确层

禁止用集成测试替代单层测试。

### 12.6 回归测试约束

以下能力必须始终有回归测试：

1. 首页输出结构
2. 认领成功链路
3. 抓取触发成功链路
4. 评分核心规则
5. 分组与公司卡排序

---

## 13. 迁移与兼容性约束

### 13.1 迁移原则

所有架构迁移必须遵守：

1. 先新增能力
2. 再双写 / 旁路
3. 再切消费方
4. 最后移除旧逻辑

禁止一次性替换整条链路。

### 13.2 旧字段兼容约束

在首页消费方完成迁移前：

1. 旧字段不得随意删除
2. 旧 DTO 契约不得破坏
3. 新能力应以附加字段或旁路快照方式落地

### 13.3 回滚约束

每个阶段性改造必须支持：

1. 回退到旧首页逻辑
2. 回退到旧评分逻辑

如果某项改造不能回滚，则不应直接进入主路径。

### 13.4 历史数据约束

改评分规则时，原则上：

1. 不重跑抓取
2. 优先重跑 analysis/backfill

系统必须支持“改规则只重算分析，不重抓全网”的方向。

---

## 14. 配置与版本化约束

### 14.1 评分规则版本化

评分结果必须支持：

1. `rule_version`
2. 可回放
3. 可比较

禁止只有当前一版规则而没有版本信息。

### 14.2 快照版本化

展示快照必须带：

1. `analysis_version`
2. `generated_at`
3. `window_start`
4. `window_end`

### 14.3 配置边界

可配置内容应包括：

1. 评分规则参数
2. 分析窗口
3. 排序开关

禁止把规则版本和硬阈值长期散落在多个 service 中。

---

## 15. 明确禁止的做法

以下做法明确禁止：

1. 在 `crawl_pipeline.py` 中继续新增首页排序规则
2. 在 `home_feed.py` 中继续新增字段识别逻辑
3. 在 API handler 中直接堆积认领业务逻辑
4. 在 crawler adapter 中直接引入评分与排序逻辑
5. 在 `Job` 一张表里无限堆积事实、分析、展示字段
6. 用前端需要的标签倒逼标准化层输出展示文案
7. 让情报层直接读原始抓取结果再独立做一套结论
8. 用集成测试代替事实提取和评分单测
9. 一次性大迁移替换旧链路
10. 改规则时用“重抓全网”代替“重算分析”
11. 在 `frontend/components/**` 中新增后端业务等级推导逻辑
12. 在 `frontend/lib/api.ts` 或页面组件中私自兼容多个未版本化 payload 变体

---

## 16. 允许的例外

以下情况允许短期例外：

1. 线上故障应急修复
2. 为保证当前首页可用而保留的兼容逻辑
3. 过渡期双写字段

但所有例外都必须满足：

1. 在代码中写清楚原因
2. 标明过渡性质
3. 有后续移除计划
4. 写明目标移除条件
5. 写明最晚移除时点或所属阶段

长期例外视为无效约束执行。

---

## 17. 约束执行清单

后续所有架构相关改动，在进入主路径前至少要回答以下问题：

1. 这段逻辑属于哪一层？
2. 它是否跨层做了不属于自己的判断？
3. 它是否违反依赖方向？
4. 它是在写事实、分析结果，还是展示快照？
5. 改它会不会反向影响其他层？
6. 它是否有对应层级的测试？
7. 它是否支持兼容与回滚？

除此之外，以下检查项必须显式通过：

1. 如果改动涉及 `home`、`claims`、`crawl/trigger` 契约，必须补回归测试
2. `crawl_pipeline.py` 不得新增 UI 语义、展示文案、页面排序规则
3. `home_feed.py` 不得新增抓取 adapter 依赖、外部请求、评分规则
4. `claims.py` 不得直接承担 claim 之外的业务编排
5. `scoring.py` 不得访问数据库，不得调用 API，不得依赖前端标签文案
6. `frontend/components/**` 不得新增后端规则判断，只能消费稳定 DTO
7. `company_daily_summary`、`intelligence_snapshot` 新增字段时，必须同步说明写入端、读取端、回滚路径
8. 所有排序、分组、等级规则必须有单独测试，测试名应能读出业务意图
9. 任何临时跨层引用，都必须先写明过渡原因和移除计划

只要其中任意一项回答不清楚，就不应直接进入主路径。

---

## 18. 一句话总结

这份约束文档的核心不是让系统“更复杂”，而是确保：

**抓取负责拿数据，事实层负责理解数据，分析层负责判断价值，展示层负责输出产品。**

任何让这些边界重新缠在一起的做法，哪怕短期看起来方便，都不应成为正式实现方向。
