# AGENTS.md

本文件是 `F:\赏金猎人` 仓库的全局协作规范。默认适用于整个仓库；如果子目录后续新增更细粒度的 `AGENTS.md`，则子目录规则优先。

## 1. 工作方式

- 默认做小而可审查的改动，不做大而散的重构。
- 改代码前，先明确要改哪些文件，并用 3-6 条列出计划。
- 不要臆造 API、配置、路径或数据字段；先查仓库再改。
- 改动必须和现有架构、命名、目录风格保持一致。
- 默认语言用中文，说明简洁、直接、可执行。

## 2. Karpathy Guidelines

以下规则来自 [$karpathy-guidelines](C:\Users\Administrator\.codex\skills\karpathy-guidelines\SKILL.md)，作为本仓库的强约束：

### 2.1 先想清楚再动手

- 明确写出假设，不要默默假设。
- 有多个解释路径时，要先说明，不要悄悄选一个。
- 如果更简单的方案可行，要优先用更简单的方案。
- 如果关键信息不清楚，先点明不清楚的地方。

### 2.2 简单优先

- 只写完成当前任务所需的最少代码。
- 不做未被要求的扩展、抽象、配置化。
- 不为单次用途提前设计通用层。
- 不写“也许以后有用”的灵活性。
- 如果一个改动明显过度复杂，必须收缩。

### 2.3 手术式修改

- 只改和当前任务直接相关的代码。
- 不顺手改邻近模块、格式、注释、命名。
- 发现无关问题可以指出，但不要顺手清理。
- 如果你的改动导致新的死代码，只清理你自己造成的部分。

### 2.4 目标驱动执行

- 每个任务都要先定义可验证的完成标准。
- 能写测试就先写失败测试，再修实现。
- 多步骤任务必须按步骤推进，并给出每一步的验证方式。
- 没有验证，不算完成。

## 3. 低耦合原则

- API handler 只做参数校验、调用 service、返回响应。
- 业务逻辑放在正确的 service / domain 边界，不堆到 API 或前端。
- 前端不承担后端业务判断。
- 不做“先 patch 一下再说”的补血式修复。
- 不用临时字段、临时兼容层、临时兜底把结构问题糊过去。

## 4. 安全与密钥

- 不要把 token、密钥、`.env` 值、数据库密码写进代码、日志或文档。
- 如果任务需要密钥，要求用户通过环境变量提供。
- 不要自行添加分析、埋点、遥测或额外网络调用。

## 5. 质量要求

- 有测试的地方，行为改动必须补或改测试。
- 优先显式错误处理和类型安全。
- 注释只写非显然意图，不写废话注释。
- 改动前优先跑最快相关验证；改完后至少跑最小相关验证。

## 6. 输出要求

### 改代码时

- 先给短摘要。
- 列出修改文件。
- 说明为什么这些修改放在当前层级是合理的。
- 说明跑了什么验证。

### 排查问题时

- 先给假设和证据。
- 明确实验/检查步骤。
- 先查根因，再提修复。

## 7. 构建与运行

- 运行命令前，先说明要跑什么、为什么。
- 优先最小验证：
  - 后端先跑最相关 `pytest`
  - 前端先跑最相关测试或 `build`
- 除非确认必要，不要一上来跑全量重任务。

## 8. 提交规范

- 一个明确任务一个 commit。
- 不把无关改动混在一个提交里。
- commit message 应清楚描述目的，而不是泛泛而谈。

## 9. 默认偏好

- 默认按“先查清、再小改、再验证、再提交”的顺序工作。
- 任何时候如果发现“代码能跑，但边界被打穿”，优先修边界，而不是继续堆逻辑。
